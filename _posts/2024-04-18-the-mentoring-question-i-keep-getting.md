Three tutoring students have asked me some version of the same question in the past
month: "How do I get better faster?" I used to answer with a reading list. I've
stopped doing that. Reading lists are what you give someone when you want to feel
like you helped without doing the harder thing, which is diagnosing what specifically
is slowing them down.

The slower students I work with aren't slower because they lack information. They're
slower because they skip the friction. They read the error message, don't understand
it, and immediately search for a solution. The faster path — embarrassingly — is to
sit with the error message and decode it before searching. What did the system
actually say? Where in the code is the line number pointing? What's the simplest
explanation consistent with the output? That process takes longer the first time and
becomes reflexive by the tenth.

The thing that actually makes people better faster is deliberate discomfort: writing
code to a spec without looking at documentation first, then checking. Predicting what
a program will print before running it. Explaining a solution out loud to someone
else, which surfaces every gap in the explanation. These feel slower in the session.
They're faster across a month. I've started ending each tutoring session with one
question: "What would you do differently next time?" A student who answers that
question well doesn't need me for long.
