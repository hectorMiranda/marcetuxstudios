I took a contract in June that I didn't scope carefully enough and I've been paying
for it since. The initial description was "integrate our data pipeline with a new
vendor API." I said yes based on that description. Three weeks in, the vendor API
was undocumented in the places that mattered, the existing pipeline had no tests, and
"integrate" turned out to include migrating the old integration's data format. None
of this was hidden — it was all visible if I'd asked the right questions before
signing the statement of work.

The questions I should have asked: what's the state of the current integration's
code and test coverage? Has this vendor API been used by anyone else you know? What's
the definition of done beyond "the data flows"? Those three questions would have
changed the scope and the price, or would have gotten me real answers that let me
budget correctly. Instead I trusted the description and absorbed the surprises.

What I've added to every scoping conversation since: a "what do you know that might
make this harder than it sounds?" question. It's disarming enough that clients
answer honestly rather than defensively. The answer is almost always useful. Sometimes
it reveals blockers that change the whole proposal. Sometimes it's nothing — and even
then you've signaled that you're thinking about the full problem. Scope creep starts
in the conversation before the contract. That's where to stop it.
